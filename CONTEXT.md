# Namahage Manager — プロジェクトコンテキスト

## 概要

マネージャー業務（欠席管理・振り返り管理）を自動化するSlack Botシステム。

### 技術スタック

| レイヤー | 技術 |
|---|---|
| バックエンド | Python（Cloud Run） |
| スケジューラ | Cloud Scheduler |
| データストア | Google Spreadsheet（Sheets API v4） |
| フロントエンド | Slack Bot API（xoxb- トークン） |
| ホスティング | GCP（Cloud Run） |

> GASは採用しない。スケールしにくい・テスト不可・コールドスタートがSlack interactivity（3秒制約）と相性が悪い・ファイル管理が困難なため。

---

## 機能一覧

### 機能1: 欠席・遅刻リマインド

- 欠席・遅刻の届け出は **Slackモーダル** で受け付ける（Google Form は廃止）
- ボタンは **欠席通知メッセージ（講義開始5時間前の投稿）に添付**する
- モーダル送信 → Cloud Run が **欠席届シートに直接書き込む**
- デフォルトは出席扱い
- 講義開始5時間前にSlackへ自動投稿：
  - オープンチャンネル: 欠席/遅刻者の名前のみ（理由なし）
  - クローズドチャンネル: 理由・詳細も含む
- 通知以降の急な欠席: 同投稿のボタンから追加提出
- ポーリング: 欠席通知投稿後〜講義開始まで、新着回答をキャッシュシートに追記

### 機能2: 振り返りリマインド

- ボットが振り返りチャンネルに親メッセージを投稿
- 部員は **スレッド返信** で振り返りを提出（タイムライン感・リアクション文化を重視するためモーダル化しない）
- 必須項目（共通）:
  - 【学んだこと】
  - 【今後、どのように活かしていくか】
  - 【自由記述】
- `conversations.replies` でスレッド返信を取得し、投稿済み判定
- 当日24:00に未投稿者へ @メンションでリマインド

## Slackモーダル仕様（欠席・遅刻届け）

```
┌─────────────────────────────┐
│  欠席・遅刻届                │
├─────────────────────────────┤
│  対象日付                    │
│  [日付ピッカー ▼]            │
│                              │
│  種別                        │
│  ○ 欠席  ○ 遅刻             │
│                              │
│  到着予定時刻（遅刻の場合）  │
│  [時刻入力]                  │
│                              │
│  理由                        │
│  [テキストエリア          ]  │
│                              │
│              [キャンセル][送信]│
└─────────────────────────────┘
```

- 送信者のSlackIDはSlackペイロードから自動取得（名前入力不要）
- 「遅刻」選択時のみ到着予定時刻フィールドを表示（Block Kit の `dispatch_action` で動的制御）

## Google Spreadsheet 構成

### シート一覧

| シート名 | 用途 |
|---|---|
| 活動カレンダー | 活動日・活動時間の記録（トリガー時刻の算出元） |
| 部員マスタ | 全部員の氏名・SlackID・チーム・除外フラグ |
| キャッシュ | 当日の欠席者・振り返り済み部員を一時管理 |
| 欠席届 | モーダル送信内容の永続記録（Google Form廃止により新設） |
| ボット本体 | Slack投稿設定（ts・チャンネルID・投稿日時など） |

### 活動カレンダー カラム構成

| 列 | 内容 | 備考 |
|---|---|---|
| A | 連番 | 活動回数 |
| B | 日付 | datetime形式 |
| C | 曜日 | 月・木など |
| D | コマ番号 | ①②③（1活動日に複数行） |
| E | テーマ | 講義名・活動内容 |
| F | 場所 | OIホールなど |
| G | 担当 | 担当者氏名 |
| H | 備考 | 特殊な注意事項 |
| I | 活動時間 | `17:30-21:00` 形式。①行にのみ記入 |

- B列で当日日付を検索し、I列に時間が入っている行を講義日の基準行とする
- I列が空白の日はトリガーを設定しない

### 部員マスタ カラム構成

| 列 | 内容 | 備考 |
|---|---|---|
| A | 名前 | 氏名 |
| B | SlackID | 例: U08RAG25KFS |
| C | チーム | 所属チーム（任意） |
| D | 除外フラグ | TRUE で集計対象外（マネージャー等） |

### キャッシュシート カラム構成

| 列 | 内容 |
|---|---|
| A | 当日欠席/遅刻者のSlackID |
| B | 振り返り済み部員のSlackID |

- 翌日12:00に `clearContents()` で全リセット

### 欠席届シート カラム構成

| 列 | 内容 |
|---|---|
| A | タイムスタンプ（送信日時） |
| B | SlackID（送信者から自動取得） |
| C | 対象日付 |
| D | 種別（欠席 / 遅刻） |
| E | 到着予定時刻（遅刻の場合のみ） |
| F | 理由 |

### ボット本体シート カラム構成（未確定、要整理）

- Slack投稿の `ts`（タイムスタンプ）
- チャンネルID
- 投稿日時

## GASトリガー → Cloud Scheduler + Cloud Run 対応表

| 旧GASトリガー | 新構成 |
|---|---|
| 毎朝07:00 固定トリガー | Cloud Scheduler → `/check-schedule` |
| 欠席通知（講義開始-5時間） | `/check-schedule` 内で Cloud Scheduler ジョブを動的登録 |
| ポーリング（定期） | Cloud Scheduler ジョブ（定期）→ `/poll-absence` |
| 振り返り投稿（講義終了時刻） | Cloud Scheduler ジョブ（動的）→ `/post-reflection` |
| リマインド（24:00） | Cloud Scheduler ジョブ（動的）→ `/remind-reflection` |
| キャッシュリセット（翌日12:00） | Cloud Scheduler ジョブ（動的）→ `/reset-cache` |

## 処理フロー

### トリガー全体フロー

```
【毎朝 07:00】Cloud Scheduler → /check-schedule
  ↓ 活動カレンダーのB列で当日日付を検索
  ↓ I列が空白 → 講義なし、何もしない
  ↓ I列に時間あり → 以下のCloud Schedulerジョブを動的登録
    1. /notify-absence     （講義開始 − 5時間）
    2. /poll-absence       （欠席通知後〜講義開始まで、定期）
    3. /post-reflection    （講義終了時刻）
    4. /remind-reflection  （当日 24:00）
    5. /reset-cache        （翌日 12:00）
```

### 欠席者の記録フロー（新）

```
部員がSlackのボタンをクリック
  ↓ Slackがinteractivity payloadをCloud Runに送信
  ↓ Cloud Runがviews.openでモーダルを表示
  ↓ 部員が入力して送信
  ↓ Slackがview_submission payloadをCloud Runに送信
  ↓ Cloud Runが欠席届シートに書き込む
  ↓ キャッシュシートA列にSlackIDを追記
```

### 振り返り未投稿者の算出

```
部員マスタ（D列がFALSEまたは空白）のSlackID
  − キャッシュシートA列（当日欠席者）
  − キャッシュシートB列（振り返り済み）
= 振り返り未投稿者 → 24:00に @メンションでリマインド
```

## 使用するSlack API

| API | 用途 |
|---|---|
| `chat.postMessage` | 欠席通知・振り返り投稿・リマインド送信 |
| `conversations.replies` | 振り返りスレッドの返信一覧を取得 |
| `views.open` | モーダルを開く |

- 認証: Bot Token（`xoxb-`）
- HTTPクライアント: Python `slack_sdk`

## 未解決事項

### 無断欠席 vs 振り返り未記入の区別問題（保留中）

| 状態 | フォーム/モーダル提出 | 振り返り投稿 |
|---|---|---|
| 正常出席 | なし | あり |
| 有断欠席 | あり（欠席） | なし |
| 有断遅刻 | あり（遅刻） | あり or なし |
| 無断欠席 | なし | なし |
| 振り返り未記入 | なし | なし ← 無断欠席と区別不能 |

→ ルール側の整備が必要なため保留

### 退部ロジック（3回ルール）

- 無断欠席または振り返り未記入を3回で退部措置
- 追跡ロジックは未設計

## マネージャーの残タスク（自動化後）

自動化完了後にマネージャーが手動で行うのは以下のみ：

1. 活動カレンダーのI列に活動時間を入力（例: `17:30-21:00`）
2. 振り返り投稿内容の確認
3. 欠席・遅刻の詳細確認

それ以外はすべて自動化。
